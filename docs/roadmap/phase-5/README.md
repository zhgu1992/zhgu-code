## 超越基线讨论（必填）

> 原则：对标源码只定义底线，不定义上限。

### 1) 超越方向

1. 门禁一致性优先：本地与 CI 使用同一门禁命令集。
2. 回归阻断优先：关键链路失败必须阻断合并。
3. 发布可追溯优先：每次发布都可回放验证证据。

### 2) 超越指标

1. `build/type/lint/test` 在 CI 中 100% 作为必过检查。
2. 回归发现提前率提升（PR 阶段识别率目标 >= 95%）。
3. flaky case 占比降至 < 2%（滚动统计）。
4. Phase 5 新增测试/脚本 case 至少 20 条，且不引入前置阶段回归。

### 3) 创新工作轨

1. SX5-1 质量门分层执行：快速门（PR）+ 全量门（release）双层策略。
2. SX5-2 验证证据归档：构建、测试、扫描结果统一归档可检索。

### 4) 风险与回滚策略

1. 门禁过严影响迭代速度时，可按层级临时降级但必须留痕。
2. 发布失败时可一键回退到最近稳定构建并保留失败证据。

## 阶段完成标准（DoD）

1. 本地与 CI 共享同一质量门命令与判定标准。
2. 测试分层清晰，网络相关测试具备可控隔离策略。
3. 安全扫描纳入发布前置门并有可追溯报告。
4. 发布前验证与回滚流程可演练、可复用。
5. `build/type/lint/test/security` 全绿并输出阶段收口报告。

## 工作包（Work Packages）

### WP5-A：质量门统一（对应 `wip5-02`）

- 目标：统一质量门命令与执行顺序。
- 产出：
  - `scripts/quality-gate.*`
  - CI workflow 中统一门禁入口
  - `src/__tests__/phase5_quality_gate.test.ts`
- 验收：本地与 CI 同命令同结果。

#### WP5-A 设计核心（必须先达成共识）

1. 为什么做（Why）
- 质量门不一致会放大“本地通过但合并失败”的协作成本。

2. 问题与边界
- In Scope：构建、类型、lint、测试、安全扫描的执行顺序与阻断规则。
- Out of Scope：业务功能逻辑修改。

3. 核心设计
- 两层门禁：`quick-gate`（PR）与 `release-gate`（发布）。
- 阻断规则：关键失败直接 non-zero 退出。
- 证据输出：每次门禁运行产出可追溯报告。

4. 验证 Case（DoD）
- `QGT-001` 快速门在 PR 环境可稳定运行。
- `QGT-002` 发布门包含完整校验项。
- `QGT-003` 任一关键步骤失败会阻断流程。
- `QGT-004` 本地与 CI 同输入同结果。
- `QGT-005` 报告包含失败模块与建议动作。
- `QGT-006` 回归场景可复现并被门禁拦截。

5. 风险与回滚
- 门禁过严导致效率下降时，允许短期降级并记录豁免。

### WP5-B：测试分层与隔离（对应 `wip5-03`）

- 目标：建立稳定测试分层与网络隔离策略。
- 产出：测试标签、分层命令、隔离说明文档。
- 验收：分层执行稳定且失败定位清晰。

### WP5-C：安全门接入（对应 `wip5-04`）

- 目标：把安全扫描纳入统一质量门。
- 产出：扫描脚本、阈值策略、失败处理流程。
- 验收：安全风险可阻断并可追溯。

### WP5-D：发布验证与回滚（对应 `wip5-05`）

- 目标：标准化发布前验证与回滚流程。
- 产出：release checklist、rollback runbook、演练记录。
- 验收：发布失败可快速回退。

### WP5-E：阶段收口（对应 `wip5-06`）

- 目标：形成质量运营闭环与长期维护策略。
- 产出：阶段报告、指标看板入口、主路线图更新。
- 验收：Phase 5 专项门禁全绿。
