# Phase 1 - Query Engine v2：给读者的核心总结

> 读者定位：想理解这套系统“为什么这样设计”的人。  
> 非目标：逐函数实现细节、逐行 API 文档。

## 1. Phase 1 到底完成了什么

Phase 1 的本质不是“多做几个功能”，而是把 Query 引擎从“能跑”升级成“可治理”。

这里的“可治理”指 4 件事：
1. 可判定：系统状态明确，不靠猜。
2. 可恢复：错误有分类，有策略，不是全靠兜底报错。
3. 可回放：会话事实可重建，不是只看实时输出。
4. 可断言：正确性可自动验证，不靠人工肉眼看日志。

## 2. 核心功能解决了哪些问题

### 2.1 显式 Turn 状态机

核心能力：`idle/streaming/awaiting-permission/tool-running/recovering/stopped`。

解决的问题：
1. 过去流程是隐式控制流，异常路径难推断。
2. 非法状态跳转不容易被及时发现。
3. 出现失败时，无法明确“为什么停在这里”。

现在的收益：
1. 每次迁移都是显式事件。
2. 终态必须有 reason（如 `completed`、`budget_exceeded`）。
3. 非法迁移可直接阻断并暴露。

---

### 2.2 主循环职责拆分（runner / consumer / orchestrator）

核心能力：把主链路从单函数拆成三个职责层。

解决的问题：
1. 单函数过重，改一处容易连锁回归。
2. budget/recovery/transcript 接线位置混乱。
3. 难以对单一环节做聚焦测试。

现在的收益：
1. 生命周期、流消费、工具编排边界清晰。
2. 对外行为保持兼容，对内可演进。
3. 后续扩展（Phase 2+）接入成本更低。

---

### 2.3 Budget Guard（预算守卫）

核心能力：在 `preflight`、`streaming`、`done` 三段做预算检查。

解决的问题：
1. token/context 过去偏“统计信息”，不是运行控制。
2. 长会话可能拖到失控才暴露问题。

现在的收益：
1. 预算从“观测指标”变成“控制面”。
2. 超限行为可预测（首版采用 stop-only）。
3. 预算触发可观测（有 trace 证据）。

---

### 2.4 错误分类 + 恢复矩阵

核心能力：先分类，再决策（retry/stop/fatal），并限制重试预算。

解决的问题：
1. 散落 `try/catch` 会造成策略漂移。
2. 同类错误在不同路径行为不一致。
3. 工具重试存在副作用风险。

现在的收益：
1. 错误路径“可判定”，不是“实现细节”。
2. provider/tool 的恢复策略分层。
3. 引入幂等保护（`safe_to_retry`）降低重复副作用风险。

---

### 2.5 最小 Transcript + Replay

核心能力：以 JSONL 记录最小事实账本，支持主链路复原。

解决的问题：
1. trace 偏诊断，不等价于“用户可核对事实”。
2. 没有统一事实账本时，复盘口径容易漂移。

现在的收益：
1. 能复原 `input -> tool_use/tool_result -> output`。
2. 不完整链路会被显式标记为 partial，不会静默吞掉。
3. 与 trace 分工明确，避免双系统重复建模。

---

### 2.6 Trace 语义断言 + 门禁命令

核心能力：把状态迁移规则写成可执行断言，并支持文件回放校验。

解决的问题：
1. “有日志”不代表“语义正确”。
2. 人工看 trace 容易漏链路断裂。

现在的收益：
1. 迁移链正确性可自动判定。
2. 可以作为门禁（`trace:assert`），而不是事后分析。
3. transcript 回放与 trace 断言形成证据闭环。

## 3. Phase 1 的核心设计思路（最重要）

这阶段最关键的不是具体代码，而是设计原则：

1. **先定义状态与不变式，再写功能分支**。  
   否则功能会增长，行为却不可验证。

2. **把“策略”从实现细节里抽出来**。  
   预算策略、恢复策略都应可判定、可替换。

3. **把“可观察”升级为“可证明”**。  
   trace 不只是看日志，而是用于自动断言语义。

4. **严格做边界分工，避免系统职责污染**。  
   transcript 记录事实，trace 记录执行诊断，各司其职。

5. **先保守，再扩展**。  
   首版先 stop-only、保守重试，保证可控性，再逐步增加策略复杂度。

## 4. 哪些地方超越了 Claude Code，对读者意味着什么

> 这里说的“超越”，是指在本仓库当前 Phase 1 的架构治理层面，做到了更显式、更可验证的工程化约束。

1. 从“可看日志”升级到“可断言语义”。
- 为什么更好：把正确性从经验判断变成自动门禁，回归效率更高。

2. 从“预算展示”升级到“预算控制”。
- 为什么更好：长会话更稳定，失败更可预测。

3. 从“错误兜底”升级到“恢复矩阵”。
- 为什么更好：行为一致性更强，异常路径更容易复盘。

4. 从“单一观测视角”升级到“trace + transcript 双证据”。
- 为什么更好：既能看执行过程，又能核对会话事实，复盘更完整。

## 5. 对学习者最值得带走的方法论

如果你在自己项目里只带走一件事，建议是这条链：

**状态机（可判定） -> 策略层（可控制） -> 事实账本（可回放） -> 断言门禁（可证明）**

这条链是 Phase 1 真正的价值，也是后续阶段可持续演进的基础。
