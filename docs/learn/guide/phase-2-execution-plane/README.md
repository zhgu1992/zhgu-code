# Phase 2 - Execution Plane：给读者的核心总结

> 读者定位：想理解系统“如何把工具执行做成可治理平面”的人。  
> 非目标：逐函数实现细节、逐行 API 文档。

## 1. Phase 2 到底在解决什么

Phase 2 的本质不是“多接几个工具”，而是把工具调用从“能执行”升级成“可治理、可追责、可回滚”。

这里的“可治理”指 4 件事：
1. 可决策：规则与风险在平台层统一判定，不靠分散分支。
2. 可解释：每次 allow/ask/deny 都有结构化原因。
3. 可追溯：单次 tool call 能还原请求、决策、执行、结果。
4. 可回退：新治理链路可开关灰度，异常可回到旧路径。

## 2. 核心能力解决了哪些问题

### 2.1 权限规则引擎（`deny > ask > allow`）

核心能力：统一规则匹配与优先级（scope/source/action）。

解决的问题：
1. 规则散落在执行器和工具层，行为容易分叉。
2. 同类场景在不同入口可能得到不同判定。
3. 默认策略不统一，导致边界不稳定。

现在的收益：
1. 决策语义收敛到平台层。
2. 冲突规则有稳定优先级。
3. 无命中规则时行为可预测（默认 ask）。

---

### 2.2 风险分级模型（静态基线 + 动态升级）

核心能力：统一计算 `ToolRiskLevel`，并根据输入特征升级风险。

解决的问题：
1. 过去只按工具名粗分，忽略输入差异。
2. 高风险输入无法在决策前被放大处理。
3. 风险结果不可复盘，难解释“为什么拒绝”。

现在的收益：
1. 风险判定可重复、可断言。
2. 高危 shell/file/network 输入可自动升级。
3. 决策链路具备稳定 reasonCode 依据。

---

### 2.3 执行入口治理接入（`risk -> permission -> mode_gate -> execute/deny`）

核心能力：在不改执行接口的前提下接入完整治理流水线。

解决的问题：
1. `permissionMode` 粗分支无法解释决策来源。
2. `auto/ask/plan` 行为一致性不足。
3. 拒绝结果偏自然语言，机器难消费。

现在的收益：
1. 模式门控语义统一。
2. 拒绝语义结构化（`reasonCode + userMessage`）。
3. 与恢复层和 UI 的契约更稳定。

---

### 2.4 审计事件链路（Audit）

核心能力：把单次 tool call 固化为可恢复证据链。

解决的问题：
1. 只有 trace 难做离线问责。
2. 不能稳定回答“为何允许/拒绝”。
3. 执行失败与决策失败容易混在一起。

现在的收益：
1. 可按 `requestId` 重建完整链路。
2. 决策与执行责任边界清晰。
3. 审计写入失败可降级，不阻断主链路。

---

### 2.5 高风险边界硬化（Bash / File / Network）

核心能力：对高危路径建立默认保守策略。

解决的问题：
1. 副作用工具缺少统一阻断边界。
2. 越界写入和危险命令风险高。
3. auto 模式下误放行代价大。

现在的收益：
1. 高风险默认不“静默放行”。
2. 拒绝路径可解释、可追溯。
3. 回滚策略明确（ask-only 或旧路径）。

## 3. Phase 2 的核心设计思路（最重要）

1. **先统一决策平面，再接执行平面**。  
   不先收敛规则与风险，执行接线只会放大分叉。

2. **机器可判定优先于人工可理解**。  
   先冻结 `reasonCode`，再扩展文案。

3. **默认保守，逐步放宽**。  
   先把误放行降到最低，再考虑体验优化。

4. **治理链路必须可回退**。  
   每个关键点都要有 feature flag 和降级路径。

5. **证据链分层**。  
   trace 用于运行诊断，audit 用于决策问责。

## 4. 截至 2026-04-10 的阶段状态

1. 已完成：规则引擎（`wip2-02`）、风险模型（`wip2-03`）、阶段收口与门禁基线（`wip2-07`）。
2. 进行中/待完成：执行入口治理接入（`wip2-04`）、审计链路（`wip2-05`）、边界硬化（`wip2-06`）。
3. 当前关注点：在不破坏 Phase 1 前置门的前提下，完成治理链路接线并保持可回滚。

## 5. 对学习者最值得带走的方法论

如果只带走一条链，建议是：

**风险建模（可分级） -> 规则决策（可判定） -> 模式门控（可执行） -> 审计证据（可追责） -> 开关回退（可治理）**
